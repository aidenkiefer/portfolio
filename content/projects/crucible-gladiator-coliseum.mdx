## Overview

Crucible is a competitive 1v1 arena combat game built to prove three hypotheses: the combat loop is fun, NFT ownership can integrate cleanly into gameplay, and real-time multiplayer can run smoothly in the browser without compromising on anti-cheat fundamentals.

It’s a full-stack TypeScript monorepo: Next.js frontend (Canvas 2D rendering), a standalone Node/Express game server running authoritative combat, a shared deterministic physics/combat package used in both environments, Solidity contracts for on-chain ownership, and Supabase/Postgres persistence.

## Problem & Context

Many blockchain games lead with token economics and ship shallow gameplay. Crucible flips that priority: build a real multiplayer combat system first, then layer on-chain ownership in a way that doesn’t disrupt the player experience.

## Constraints

- Web-first (Canvas 2D) at 60 FPS.
- Testnet-only for the demo.
- Server-authoritative outcomes; client can predict but never decides results.
- Small team and sprint-bounded scope.

## Approach & Design Decisions

- **Shared deterministic simulation**: a pure TypeScript physics/combat package runs identically in Node and the browser.
- **60 Hz simulation, 20 Hz broadcast**: the server ticks continuously and broadcasts compressed snapshots; the client predicts locally and reconciles.
- **Separate long-running server**: real-time match state lives in a dedicated game server, not a serverless function.
- **Template/instance content model**: equipment templates are authored in an admin UI and published as bundles so runtime avoids DB calls mid-match.

## Implementation Highlights

- Weapons + projectiles with server-side validation.
- Input validation + rate limiting for anti-cheat.
- Matchmaking, disconnect/reconnect recovery, and persistent match history.
- Smart contract integration (ERC-721) with event syncing.
